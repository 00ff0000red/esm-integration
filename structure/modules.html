Modules¶

A module collects definitions for types, functions, tables, memories, and globals. In addition, it can declare imports and exports and provide initialization logic in the form of data and element segments or a start function.

\[\begin{split}\begin{array}{lllll} \def\void#1{}\void{(modules)} & \mathit{module} &::=& \{ & \mathsf{types}~vec(\href{types.html#syntax-functype}{\mathit{functype}}), \\&&&& \mathsf{funcs}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-func}{\mathit{func}}), \\&&&& \mathsf{tables}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-table}{\mathit{table}}), \\&&&& \mathsf{mems}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-mem}{\mathit{mem}}), \\&&&& \mathsf{globals}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-global}{\mathit{global}}), \\&&&& \mathsf{elem}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-elem}{\mathit{elemseg}}), \\&&&& \mathsf{data}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-data}{\mathit{dataseg}}), \\&&&& \mathsf{start}~\href{#indices}{\href{#syntax-start}{\mathit{funcidx}}}, \\&&&& \mathsf{imports}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-import}{\mathit{import}}), \\&&&& \mathsf{exports}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{#syntax-export}{\mathit{export}}) \quad\} \\ \end{array}\end{split}\]

Indices¶

Definitions are referenced with zero-based indices. Each class of definition has its own index space, as distinguished by the following classes.

\[\begin{split}\begin{array}{llll} \def\void#1{}\void{(type indices)} & \mathit{typeidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(function indices)} & \mathit{funcidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(table indices)} & \mathit{tableidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(memory indices)} & \mathit{memidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(global indices)} & \mathit{globalidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(local indices)} & \mathit{localidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \def\void#1{}\void{(label indices)} & \mathit{labelidx} &::=& \href{#numbers}{{\mathit{uint}_{32}}} \\ \end{array}\end{split}\]

The index space for functions, tables, memories and globals includes respective imports declared in the program. The indices of these imports precede the indices of other definitions.

The index space for locals is only accessible inside a function and includes the parameters and local variables of that function.

Label indices reference block instructions inside an instruction sequence.

Types¶

The \(\mathsf{types}\) of a module define a vector of function types.

All function types used in a module must be defined in the type section. They are referenced by type indices.

Note

Future versions of WebAssembly may add additional forms of type definitions.

Functions¶

The \(\mathsf{funcs}\) of a module declare a vector of functions defined as follows:

\[\begin{split}\begin{array}{llll} \def\void#1{}\void{(functions)} & \mathit{func} &::=& \{ \mathsf{type}~\mathit{typeidx}, \mathsf{locals}~\href{../structure/values.html#syntax-vec}{\mathit{vec}}(\href{types.rst#syntax-valtype}{\mathit{valtype}}), \mathsf{body}~\mathit{expr} \} \\ \end{array}\end{split}\]

The \(\mathsf{type}\) of a function declares its signature by reference to a type defined in the module. The parameters of the function are referenced through 0-based local indices in the function’s body.

The \(\mathsf{locals}\) declare a vector of mutable local variables and their types. These variables are referenced through local indices in the function’s body. The index of the first local is the smallest index not referencing a parameter.

The \(\mathsf{body}\) is an instruction sequence that must evaluate to a stack matching the function type’s result type.

Tables¶

The \(\mathsf{tables}\) of a module define a vector of tables described by their table type:

\[\begin{split}\begin{array}{llll} \def\void#1{}\void{(tables)} & \mathit{table} &::=& \{ \mathsf{type}~\mathit{tabletype} \} \\ \end{array}\end{split}\]

A table is a vector of opaque values of a particular table element type. The initial size of each table is given by the \(\mathsf{min}\) size specified in the limits of its table type. Each table may be grown dynamically, but only up to its \(\mathsf{max}\) size if specified.

Tables can be initialized through element segments.

Tables are referenced through table indices. Most constructs implicitly reference table index \(0\).

Note

Currently, at most one table may be defined or imported in a single module, and all constructs implicitly reference this table \(0\). This restriction may be lifted in future versions of WebAssembly.

Tables can contain values that are not otherwise accessible – like host object references, raw OS handles, or native pointers – so that they can be accessed indirectly through an integer index. That bridges the gap between low-level, untrusted linear memory and high-level opaque handles or references.

Currently, the primary purpose of tables is to emulate function pointers, which can be represented as integers indexing into a table of type \(\mathsf{anyfunc}\) holding functions and can be called via the \(\mathsf{call\_indirect}\) instruction.

Memories¶

The \(\mathsf{mems}\) of a module define a vector of linear memories as described by their memory type:

\[\begin{split}\begin{array}{llll} \def\void#1{}\void{(memories)} & \mathit{mem} &::=& \{ \mathsf{type}~\mathit{memtype} \} \\ \end{array}\end{split}\]

A memory is a vector of raw uninterpreted bytes. The initial size of each memory is given by the \(\mathsf{min}\) size specified in the limit of its table type. Each memory may be grown dynamically in units of page size, but only up to its \(\mathsf{max}\) size if specified.

Memories can be initialized through data segments, but are otherwise initialized with zero bytes.

Memories are referenced through memory indices. Most constructs implicitly reference memory index \(0\).

Note

Currently, at most one memory may be defined or imported in a single module, and all constructs implicitly reference this memory \(0\). This restriction may be lifted in future versions of WebAssembly.

It is unspecified how embedders map this array into their process’ own virtual memory. However, linear memory is sandboxed and does not alias other memory regions.
